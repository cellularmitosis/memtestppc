# memtest86+ v1.00 vs v2.00 — port-base comparison

Comparison of the two extracted candidate sources to pick the base for the
PowerPC/Open-Firmware port (memtestppc+).

- **v1.00** (Dec 2003): `ref/memtest86+-1.00/` — 10,112 lines total
- **v2.00** (Feb 2008): `ref/memtest86+-2.00/` — 14,364 lines total
- (reference only, NOT a candidate) **v5.01**: `ref/memtest86+-5.01/` — heavily multi-CPU

Decision constraints (from the brief): single-CPU only; verbatim-copy-then-comment
porting discipline (never delete upstream, so *less x86/BIOS/chipset cruft = cleaner*);
faithful reproduction of the chosen version's exact TUI; display reaches the screen via
a virtual VGA text buffer (80x25 char+attr at the `0xb8000` stand-in) blitted to the
framebuffer.

**Bottom line up front: recommend v1.00.** Both are equally single-CPU (zero SMP code),
but v1.00 carries far less x86/BIOS/chipset cruft (no `dmi.c`, no `spd.c`, no `extra.c`,
a controller.c half the size), its error path is simpler, and its TUI is the classic
blue screen with only cosmetic differences from v2.00. The one real thing v2.00 buys you
— a richer "summary"/confidence error screen and 3 extra test patterns — is not worth the
~4,250 extra lines of PC-chipset leaf code you'd have to wade through and comment out.
Details and the honest tradeoffs below.

## 2. Error-reporting & error-table path  ← KEY DECISION POINT

### v1.00 — error path lives in `test.c` (no `error.c`)

All reporting is at the bottom of `ref/memtest86+-1.00/test.c:1118-1424`:

- `error()` (`test.c:1121`) — data-error entry; dedup against last `(adr,xor)`, then
  `print_err()` or BadRAM-pattern path depending on `v->printmode`.
- `mv_error()` (`test.c:1166`) — block-move test error (failing address unknown).
- `ad_err1()` (`test.c:1183`) / `ad_err2()` (`test.c:1199`) — address-test errors.
- `print_hdr()` (`test.c:1216`) — lazily prints the two header rows once.
- `update_err_counts()` (`test.c:1227`) — `++v->ecount; tseq[v->test].errors++;` (4 lines).
- `print_err_counts()` (`test.c:1233`) — repaints the two count fields.
- `common_err(page, offset)` (`test.c:1239`) — scrolls, prints Tst/Pass/address columns.
- `print_err()` (`test.c:1263`) — fills Good/Bad/Err-Bits/Count.
- `print_ecc_err()` / `parity_err()` (`test.c:1283` / `1304`) — ECC/parity rows.
- `printpatn()` (`test.c:1326`) — emits the `badram=0x..,0x..` LILO option line.
- `do_tick()` (`test.c:1361`) — progress bars + WallTime (rdtsc).

There is exactly **one** error screen — the scrolling address table — drawn by `print_hdr()`:

```
Tst  Pass   Failing Address          Good       Bad     Err-Bits  Count Chan
---  ----  -----------------------  --------  --------  --------  ----- ----
```

Columns (from `common_err`/`print_err`, `test.c:1239-1278`):
`Tst`@col0(3), `Pass`@col4(5), page-hex@col11, offset-hex@col19, ` -  . MB` decimal MB
readout@col22-31, `Good`@col36, `Bad`@col46, `Err-Bits`(xor)@col56, `Count`@col66.
ECC rows write "corrected/uncorrected", a 4-digit syndrome@col60, "ECC"@col68, chan@col74.

`v->printmode` in v1.00 has only **3** values (`test.h:152`): `PRINTMODE_ADDRESSES`(0,
default), `PRINTMODE_PATTERNS`(1, BadRAM), `PRINTMODE_NONE`(2).

### v2.00 — dedicated `error.c` (547 lines)

`ref/memtest86+-2.00/error.c` centralizes everything and adds a second, much richer
display mode. Entry points:

- `error()` (`error.c:31`) — data error; computes xor, calls `common_err(...,type=0)`.
- `ad_err1()` (`error.c:54`) — address error (mask), `type=1`.
- `ad_err2()` (`error.c:64`) — address-equals-own-address error, `type=0` + BadRAM.
- `print_ecc_err()` (`error.c:365`, `type=2`) / `parity_err()` (`error.c:378`, `type=3`).
- `common_err(adr,good,bad,xor,type)` (`error.c:110`) — the hub; one big `switch(printmode)`.
- `update_err_counts()` (`error.c:69`) — same `++ecount`/`tseq[].errors++` **plus** optional
  `beep(600);beep(1000)` (beepmode) and clearing the "Pass complete" message.
- `print_err_counts()` (`error.c:85`) — repaints counts **and paints the whole error row
  red** (`*pp = 0x47`, `error.c:97-104`) for ADDRESSES/PATTERNS modes — this is the "vivid
  red error row" the memtestppc+ TUI already mimics (per the project README/session 005).
- `printpatn()` (`error.c:394`) / `do_tick()` (`error.c:429`).

`v->printmode` in v2.00 has **5** values (`test.h:180`): `PRINTMODE_SUMMARY`(0),
`PRINTMODE_ADDRESSES`(1, the shipped default — `init.c:180`), `PRINTMODE_PATTERNS`(2),
`PRINTMODE_NONE`(3), `PRINTMODE_DMI`(4).

**PRINTMODE_ADDRESSES (the default — same screen as v1.00):** identical header and column
layout to v1.00 (`error.c:287-290` matches v1.00 verbatim), same MB readout, same
Good/Bad/Err-Bits/Count/Chan fields. The only functional add is the red-row repaint.

**PRINTMODE_SUMMARY (NEW, no v1.00 equivalent):** a non-scrolling statistics panel
(`error.c:120-277`) drawn at `LINE_HEADER`:

```
Error Confidence Value:
  Lowest Error Address:    <page>...  -  . MB
 Highest Error Address:    <page>...  -  . MB
    Bits in Error Mask:    <ebits hex>
 Bits in Error - Total:    Min:    Max:    Avg:
 Max Contiguous Errors:
ECC Correctable Errors:
Errors per Memory Slot:    00: 0  01: 0 ...     Test  Errors
```

It also lists per-`tseq` test error counts down the right side (col 66/68) and, if
`dmi_initialized`, per-DIMM-slot error counts wired to `dmi_err_cnts[]` (from `dmi.c`).
This summary screen depends on the `struct err_info v->erri` block (`test.h:276-288`,
added to `struct vars`) and on DMI.

**Which is cleaner to port / which is richer?**
- *Cleaner:* **v1.00.** Its error path is ~300 lines of plain `cprint`/`dprint`/`hprint`
  with no external deps (no DMI, no `erri` struct, no beep). It's the single screen we
  already reproduce.
- *Richer:* **v2.00**, by the SUMMARY screen + per-slot DMI attribution + the red-row paint.
  But the richness (confidence %, per-slot counts) is **DMI/SMBIOS-dependent**, which doesn't
  exist on a PPC Mac — the per-slot table would be dead on our hardware. The red-row paint
  is trivial to lift into a v1.00 base by hand if we want it (and per session 005 we already
  have an "error-row red" effect). Net: v2.00's error *richness* is largely PC-only.

## 3. TUI / static screen layout

Both screens are the classic 80x25 blue text screen, drawn by `display_init()` + `init()`.
The layout constants in `test.h` are **almost identical** (`COL_TST` is the only positional
difference: 52 in v1.00 `test.h:64` vs 51 in v2.00 `test.h:71`). Both share:
`LINE_CPU=1`, `COL_MID=30`, `LINE_TST=2`, `LINE_RANGE=3`, `LINE_PAT=4`/`COL_PAT=41`,
`LINE_TIME=10`/`COL_TIME=0`, `LINE_INFO=10`, `LINE_HEADER=12`, `LINE_SCROLL=14`,
`BAR_SIZE=(78-COL_MID-9)=39`.

### Title bar (row 0)

- v1.00 (`init.c:54`): `"      Memtest86  v1.00       "` over a red band (`TITLE_WIDTH=28`),
  plus a `"+"` plotted at col 15 (`init.c:59`) and two `0xA4` attribute cells — i.e. it
  renders as **Memtest86+ v1.00**.
- v2.00 (`init.c:66`): `"      Memtest86  v2.00      "`, identical `"+"` overlay → **Memtest86+ v2.00**.
- Only the version string differs.

### Right-hand progress/status block (cols ~30-78) — IDENTICAL between versions

Drawn at `init.c` (v1.00 `:120-134`, v2.00 `:162-176`), both print:

```
row 0  col30: "Pass   %"          (Pass-progress bar starts COL_MID+9=39, 39 wide)
row 1  col30: "Test   %"          (Test-progress bar, same geometry)
row 2  col30: "Test #"
row 3  col30: "Testing: "         (range "lo - hi  <selected> Relocated")
row 4  col30: "Pattern: "
rows 0-4 col28: "| "              (vertical separator, COL_MID-2)
```

### Left-hand CPU/cache block (rows 1-4, col 0) — IDENTICAL labels

```
row 1 (LINE_CPU)   : <cpu name + speed>          (e.g. "Pentium III  933 MHz")
row 2 (LINE_CPU+1) : "L1 Cache: Unknown "        -> "L1 Cache:  nnnK  nnnnnn MB/s"
row 3 (LINE_CPU+2) : "L2 Cache: Unknown "        -> "L2 Cache: nnnnK  nnnnnn MB/s"
row 4 (LINE_CPU+3) : "Memory  : <size>           nnnnnn MB/s"
row 5 (LINE_CPU+4) : "Chipset : <controller>"
```
(v2.00 nudges the L1/L2 MB/s field one col right — `init.c:990/1010` use `+16` vs v1.00
`+15` — cosmetic.) `WallTime` clock is `LINE_TIME=10, COL_TIME=0`, format `":  :"` set at
`init.c` (`v1:118` / `v2:160`), filled by `do_tick`.

### Status table (rows 8-11) — IDENTICAL between versions

```
row 8 (LINE_INFO-2): " WallTime   Cached  RsvdMem   MemMap   Cache  ECC  Test  Pass  Errors ECC Errs"
row 9 (LINE_INFO-1): " ---------  ------  -------  --------  -----  ---  ----  ----  ------ --------"
row 10(LINE_INFO)  : <values> ... "Std" @COL_TST, "    0" @COL_PASS, "     0" @COL_ERR
row 11(LINE_INFO+1): " -----------------------------------------------------------------------------"
```
(v2.00 writes `" Std"` with a leading space at `COL_TST=51`; v1.00 writes `"Std"` at
`COL_TST=52` — same glyphs land in the same cells.)

### Footer (row 24) — BYTE-IDENTICAL

Both `lib.c footer()` (v1 `:484`, v2 `:597`):
```
(ESC)Reboot  (c)configuration  (SP)scroll_lock  (CR)scroll_unlock
```
with `"LOCKED"`/blank at col 74, on a reverse-video bar (`init.c` sets attr `0x71`).

### Error/scroll region

Header at `LINE_HEADER=12`, scroll body rows 14-23. v2.00 adds `LINE_MSG=18`/`COL_MSG=18`
(`test.h:78-79`) used only for the "Pass complete, no errors, press Esc to exit" banner
(`main.c:450`) and the summary panel — v1.00 has no such banner.

### Verdict

The **static screen is effectively the same** in both — same labels, same columns, same
separators, same footer. v2.00's *only* user-visible TUI additions are: (a) the "Pass
complete…" message line, (b) the optional SUMMARY error panel, and (c) the red error-row
paint. None of these is a layout change to the main screen. So "TUI fidelity" does **not**
favor v2.00 in any structural way — a v1.00 base reproduces the same classic screen, and
the two extras (banner, red row) are a few lines to add by hand if desired.

## 4. Test suite (`tseq[]`) & `test.c`

### v1.00 tseq (`main.c:12-27`) — `DEFTESTS=7`, 8 default + 4 extended rows

| # | cache | pat | iter | msg |
|---|---|---|---|---|
| 0 | 0 | 5 | 3 | Address test, walking ones, no cache |
| 1 | 1 | 0 | 3 | Moving inv, ones & zeros, cached |
| 2 | 0 | 6 | 3 | Address test, own address, no cache |
| 3 | 1 | 1 | 2 | Moving inv, 8 bit pattern, cached |
| 4 | 1 | 2 | 2 | Moving inv, 32 bit pattern, cached |
| 5 | 1 | 7 | 64 | Block move, 64 moves, cached |
| 6 | 1 | 3 | 4 | Modulo 20, ones & zeros, cached |
| 7 | 0 | 0 | 2 | Moving inv, ones & zeros, no cache |
| 8.. | | | | (extended: 512-move block, no-cache variants, modulo-20 8-bit) |

Patterns used: 0,1,2,3 (movinv), 5,6 (addr), 7 (block move). **No random tests, no bit-fade.**

### v2.00 tseq (`main.c:18-30`) — `DEFTESTS=9`, 10 rows

| # | cache | pat | iter | msg |
|---|---|---|---|---|
| 0 | 0 | 5 | 3 | Address test, walking ones, uncached |
| 1 | 1 | 6 | 3 | Address test, own address |
| 2 | 1 | 0 | 3 | Moving inversions, ones & zeros |
| 3 | 1 | 1 | 2 | Moving inversions, 8 bit pattern |
| 4 | 1 | **10** | 60 | Moving inversions, **random pattern** |
| 5 | 1 | 7 | 64 | Block move, 64 moves |
| 6 | 1 | 2 | 2 | Moving inversions, 32 bit pattern |
| 7 | 1 | **9** | 40 | **Random number sequence** |
| 8 | 1 | 3 | 4 | Modulo 20, random pattern |
| 9 | 1 | **8** | 1 | **Bit fade test, 90 min, 2 patterns** |

**Did the test set change?** Yes, modestly. v2.00 adds three pattern kinds:
- `pat 8` → `bit_fade()` (`test.c:1289`) — writes a pattern, `sleep(5400s)`, re-checks.
- `pat 9` → `movinvr()` (`test.c:257`) — moving-inv over a PRNG stream.
- `pat 10` → random-data moving-inv (handled inline in `main.c:380`).
These need `random.c` (`rand`/`rand_seed`) and `sleep()` (which is **rdtsc-based**,
`test.c:1346`). The classic deterministic tests (addr1/2, movinv1, movinv32, modtst,
block_move) are present in **both** and are essentially the same code.

### test.c algorithm changes between the two

The leaf test routines are mostly identical x86 inline-asm. Two real diffs:
1. **`movinv32`** — v2.00 has a hand-optimized inner loop ("CDH" edits, `test.c:607-877`)
   using `roll`/`incb`/`andb $31` instead of v1.00's explicit shift+counter
   (`1.00/test.c:486-508` etc). Same algorithm, tighter asm. The original C is preserved as
   comments in both (good for our port — we re-implement from the C, not the asm).
2. **`addr_tst1`** — v2.00 steps `p += bank/4` (`test.c:119`) where v1.00 steps `p += bank`
   (`1.00/test.c:112`) — finer address-bit coverage per bank.
3. v2.00's `modtst` clears the pattern field at the end (`test.c:1066`); cosmetic.

**Which has more/better tests?** v2.00 (random + bit-fade are genuinely useful). **But** for
our port both the random tests and bit-fade depend on x86 `rdtsc` for seeding/timing
(`test.c:265-270`, `1353`), so they're extra porting work, and bit-fade's 90-minute sleep is
of marginal value on a boot-from-CD memory tester. The deterministic core — which is what
catches the overwhelming majority of faults — is identical in both.

## 5. `main.c` structure & relocation/windowing

Both are **single-CPU** with the *same* main loop and windowing machinery. `main.c` diffs
are small:

- **`tseq[]`** differs (see §4) and v2.00 uses `#define DEFTESTS 9` + a hardcoded
  `if (v->test >= 9 ...)` end-of-pass check (`main.c:441`) vs v1.00's `tseq[].msg==NULL`
  sentinel + extended/default `xtst_flag` machinery (`main.c:209-228`, removed in v2.00).
- **`parse_command_line()`** — NEW in v2.00 (`main.c:144-175`): reads the Linux boot
  protocol cmdline for `console=ttyS...` (serial console). Pure x86/BIOS; we'd comment it out.
- **`high_test_adr`** — v1.00 uses a compile-time `HIGH_TEST_ADR`; v2.00 makes it a runtime
  `ulong high_test_adr` set to 16 MB if RAM > ~17 MB (`main.c:43,202-206`). Minor.
- `pat 8` (bit_fade) runs only in `window==0` (`main.c:368`).

**Relocation / windowing (the part that matters for the OFW port):**
- `windows[]` table (`main.c:41-96` both) — same 2 MB-ish window scheme covering 0–16 MB.
- `run_at()` / `__run_at()` (`main.c:102-136` both) — `memmove` the whole image to LOW
  (`LOW_TEST_ADR`) or HIGH copy and `goto *startup_32` — **identical logic**.
- `compute_segments()` (`main.c:473-556` v1 / `523-606` v2) — **identical** (incl. the same
  `#if 0` debug blocks).
- `find_ticks()` / `find_ticks_for_test()` — **identical** except v1.00 honors
  `xtst_flag`/`cache_flag` selection (`main.c:439-459`) which v2.00 dropped.
- `reloc.c` is **byte-identical** between the two (262 lines each).

For the port, the relocate-the-whole-image-and-jump trick is the trickiest piece either
way (it presumes flat physical addressing; under OFW's MMU it interacts with our claim/map
strategy — see invariant #4 in CLAUDE.md). It is the **same difficulty in both versions**,
so this dimension is a wash — except v1.00's `main.c` is slightly simpler (no cmdline parse,
no runtime high_test_adr).

**SMP/barrier references:** `grep` for `smp|barrier|num_cpus|ap_continue|cpu_num` returns
**0 hits in both v1.00 and v2.00**. (For contrast, v5.01 hits in init.c, error.c, lib.c,
main.c, test.c, random.c, config.h, smp.h, and a dedicated `smp.c`.) Both candidates are
genuinely single-CPU — neither carries the SMP tax. This dimension does **not** distinguish
them; it only rules out v5.01.

## 6. x86 / BIOS / PC-hardware leaf-code surface (the comment-out burden)

The code we'd have to neutralize (comment out + provide OFW/PPC equivalents). Approximate
line counts of the platform-specific surface:

| Concern | File(s) | v1.00 lines | v2.00 lines |
|---|---|---:|---:|
| Memory-controller (chipset) detection | `controller.c` | 1330 | **2694** |
| DMI / SMBIOS table parse | `dmi.c`,`dmi.h` | 0 | **337** |
| SPD EEPROM over SMBus | `spd.c` | 0 | **140** |
| Memory-timing overclock menu | `extra.c`,`extra.h` | 0 | **1039** |
| BIOS memory probe (E820/E801/E88) | `memsize.c` | 563 | 515 |
| PCI config-space access | `pci.c`,`pci.h` | 229 | 255 |
| LinuxBIOS/coreboot table probe | `linuxbios.c` (+h) | 236 | 236 |
| x86 real-mode boot stub | `bootsect.S`,`setup.S` | 490 | 490 |
| Startup/GDT/IDT/paging asm | `head.S` | 921 | 984 |
| Serial console (UART) | `serial.h` + lib.c bits | ~163 | ~163 + cmdline |
| PC keyboard (i8042 `inb 0x60/0x64`) | `lib.c get_key/check_input` | ~60 | ~60 |
| `rdtsc` timing, `cache_on/off`, MSR | `test.h`,`init.c`,`msr.h` | scattered | scattered + `correct_tsc` |
| **Roughly platform-specific total** | | **~4,200** | **~6,800** |

Key observations:
- **`controller.c` nearly doubles** (1330 → 2694). Setup/poll routine count goes 36 → 61 —
  v2.00 supports many more Intel/AMD/VIA/nForce chipsets, all irrelevant on a PPC Mac. Under
  "never delete," that's ~1,360 extra lines of dead, commented chipset code to carry.
- **`dmi.c` + `spd.c` + `extra.c` (~1,516 lines) are 100% PC-platform** and have **no PPC
  analog**: DMI/SMBIOS, SMBus SPD reads, and a BIOS-poke memory-overclocking menu
  (`change_timing_i875`, `amd64_tweak`, etc.). `extra.c` is wired into `main.c`/`config.c`
  via `get_menu()`/`performance()`. All of it would be commented out wholesale.
- `init.c` is also +373 lines in v2.00, almost entirely a longer CPU-ID `switch` (Core 2,
  Nehalem, Phenom, VIA Isaiah…) plus `correct_tsc()` MSR scaling — none meaningful on PPC.
- `head.S` (+63) adds more x86 startup; we replace `head.S` with our own PPC `head.S`
  regardless, so this is moot either way.

So the comment-out burden is roughly **60% larger** in v2.00 (~6,800 vs ~4,200 lines), and
the *added* 2,600 lines are exactly the most PC-coupled, least-portable code in the tree.

## 7. Display primitives (`lib.c`) & `screen_buffer.c`

- **`screen_buffer.c`/`.h` are byte-identical** between the versions (diff is whitespace
  only). The virtual VGA buffer abstraction (`get_scrn_buf`/`set_scrn_buf`/`tty_print_*`)
  we'd reuse is the same. Note: this is the *serial-echo* mirror buffer, distinct from the
  `0xb8000` char+attr array — both versions write the live screen straight to `SCREEN_ADR`
  in `cprint` (`lib.c` `*dptr = text[i]; dptr += 2;`).
- **Core display primitives are unchanged**: `cprint`, `dprint`, `aprint`, `hprint`,
  `hprint2`, `xprint`, and `scroll` are line-for-line the same (v1 `lib.c:96-310` vs v2
  `lib.c:174-432`). The `0xb8000`→buffer write loop in `cprint` is identical.
- **v2.00 adds to `lib.c`** (additive, doesn't disturb the v1.00 primitives):
  `clear_scroll()` (`lib.c:209`, used by the summary screen), `hprint3()` (exact-width hex,
  `lib.c:383`), C-lib helpers `toupper/isdigit/isxdigit/simple_strtoul/strncmp`
  (`lib.c:92-167`), and `serial_console_setup()` (`lib.c:1000`, the cmdline serial path).
  `set_cache()` in v1.00 prints upper-case OFF/ON when cache is force-toggled (`lib.c:407`);
  v2.00 always prints lower-case (`lib.c:529`). The `inter()` x86 fault dumper and the
  `get_key`/`check_input` PC-keyboard path are identical.

**Conclusion:** the display layer we actually port is the same in both. Picking v1.00 loses
nothing here; the few v2.00 helpers (`clear_scroll`, `hprint3`) are easy to add by hand if a
later feature wants them.

## 8. Overall recommendation

### Recommend: **base the port on v1.00.**

Reasoning against the four decision axes:

1. **Single-CPU:** tie — both have zero SMP code. (This only matters for excluding v5.01.)
2. **Least x86/chipset cruft (verbatim-copy-then-comment discipline):** **v1.00 wins
   decisively.** It has no `dmi.c`/`spd.c`/`extra.c`/`random.c`, a `controller.c` half the
   size, and ~2,600 fewer lines of the most PC-coupled code. Every one of v2.00's additions
   except `error.c` is platform code we'd immediately comment out and carry as dead weight.
3. **TUI fidelity:** near-tie, slight edge to v1.00 for simplicity. The static screen is the
   same classic blue layout in both (same labels/columns/separators/footer); v2.00's only
   extras are a "Pass complete" banner, an optional DMI-dependent SUMMARY panel, and a
   red-error-row paint — none of which change the main screen, and the first/third are a few
   lines to graft onto a v1.00 base if wanted (session 005 already has an error-row-red
   effect).
4. **Tests & error screen (where v2.00 is genuinely better):** v2.00 adds random + bit-fade
   tests and a richer summary/per-slot error panel. **But** the random/bit-fade tests rely on
   x86 `rdtsc`, and the per-slot error attribution relies on DMI/SPD that don't exist on PPC
   Macs — so most of v2.00's advantage here evaporates on our target.

### Honest tradeoffs / gotchas either way

- **Choosing v1.00, you give up:** `error.c` as a clean separate module (you inherit the
  error path embedded in `test.c:1118-1424` instead — slightly less tidy but ~300 lines of
  plain prints, easy to port); the SUMMARY/confidence error screen; the red-error-row paint
  (re-add by hand); 3 extra test patterns (random×2 + bit-fade); and `clear_scroll`/`hprint3`
  helpers (re-add by hand). None of these block the mission ("faithful, trustworthy port").
- **Choosing v2.00, you pay:** ~2,600 extra lines to read and comment out — and the bulk of
  it (`controller.c` growth, `dmi.c`, `spd.c`, `extra.c`) is the densest, most BIOS-poking
  code in the project, which works *against* the project's central discipline (keep the diff
  vs upstream small and navigable). The serial-console cmdline path (`parse_command_line` +
  `serial_console_setup`) assumes the Linux x86 boot protocol and is meaningless under OFW.
- **Shared gotcha (both):** the `run_at()`/windowing relocation assumes flat physical
  addressing; under OFW's MMU it must be reconciled with the claim/map approach
  (CLAUDE.md invariant #4). Equal difficulty in both — not a differentiator.
- **If TUI fidelity to the *specific* version's screen is a hard requirement:** confirm which
  classic screenshot the project is targeting. The two reference images in `ref/screenshots`
  / `ref/photos` are a v6.00 (totally different modern layout) and a low-res classic shot;
  both v1.00 and v2.00 render the same "Memtest86+ vX.XX" classic blue screen described in §3,
  so either base reproduces "the" classic look — the version string is the only title diff.

**Net:** v1.00 is the smaller, cleaner, lower-risk base that still reproduces the classic
TUI and the full deterministic test suite. Start from v1.00 and, if later wanted, hand-port
the two or three small v2.00 niceties (red error row, "Pass complete" banner) — far cheaper
than commenting out 2,600 lines of irrelevant chipset code.
